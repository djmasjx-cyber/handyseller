### Аналитика HandySeller — дорожная карта по этапам

Этот документ фиксирует общую архитектуру и план развития раздела **«Аналитика»** для WB, Ozon и Яндекс Маркета. Его цель — чтобы любой агент/разработчик мог продолжить работу без потери контекста.

---

### 1. Цели аналитики

- **Для продавца**:
  - быстро понять, как идут продажи по каждому маркетплейсу и в целом;
  - видеть проблемные и перспективные товары;
  - получать понятные рекомендации по действиям (цена, карточка, остатки, реклама).
- **Для системы**:
  - единая модель данных для разных маркетплейсов;
  - масштабируемость по числу площадок и объёму данных;
  - минимальная нагрузка на внешние API (WB, Ozon, Яндекс).

---

### 2. Источники данных

**Внешние:**
- **Wildberries (WB API):**
  - Orders API — заказы и выручка.
  - Analytics API — воронка карточки (просмотры, корзины, заказы), поисковые запросы (позже).
  - Reports / financial — возвраты, удержания, платное хранение (позже).
- **Ozon Seller API:**
  - Analytics (`/v1/analytics/data` и аналоги) — выручка, заказы по дням.
  - Products (`/v2/product/list`) — количество активных товаров.
  - Финансовые транзакции (позже, для точной маржи).
- **Яндекс Маркет:**
  - Статистика заказов и кампаний (orders stats).
  - Offers — количество активных предложений.

**Внутренние (HandySeller):**
- Таблица заказов (единая модель для WB/Ozon/Яндекс).
- Продукты (`CanonicalProduct`) и маппинг на маркетплейсы.
- Остатки, время обработки, статусы заказов.
- (Потенциал) Себестоимость для расчёта маржи.

---

### 3. Текущая реализация (Этап 0 — сделано)

**Бэкенд:**
- В `MarketplacesAdapter` реализованы методы `getStatistics()` для:
  - **WB**: считает `totalProducts`, `totalOrders`, `revenue` за **текущий календарный месяц**.
  - **Ozon**: те же показатели за календарный месяц через `analytics/data`.
  - **Яндекс**: аналогично через `stats/orders` и `offers`.
- В `MarketplacesService.getStatistics(userId)` собирается словарь:
  - `statistics[marketplace] = { totalProducts, totalOrders, revenue, lastSyncAt }`.
- `/api/marketplaces/statistics` — отдаёт эти данные на фронт.
- `/app/api/dashboard/route.ts`:
  - вызывает `/marketplaces/statistics`,
  - считает агрегаты `totalRevenue`, `totalOrders`, `totalProductsOnMarketplaces`,
  - использует **дату с начала текущего месяца** для заказов.

**Фронтенд (вкладка `/dashboard/analytics`):**
- Создана страница `AnalyticsPage` в стиле остального дашборда:
  - верхний блок: заголовок, кнопка «Обновить».
  - **KPI‑карточки**:
    - **Выручка** за текущий месяц + разбивка: `WB: … ₽ · Ozon: … ₽ · Яндекс: … ₽`.
    - **Заказы** за месяц + разбивка по маркетплейсам.
    - **Товары на площадках** — сумма `totalProducts` из статистики + разбивка по маркетам.
  - Карточка «Площадки» **скрыта** (чтобы не вводить в заблуждение, если есть только связки, но нет реальных продаж).
  - Блок «По площадкам» — карточки по каждому маркету (WB/Ozon/Яндекс) с выручкой, заказами, количеством товаров и временем последней синхронизации.
  - Блоки‑заглушки:
    - «Рекомендации» (опишем ниже, как заполнить),
    - «Аналитика по товарам» (будет таблица товаров/воронок).

### 4. Этап 1 — Завершение базового обзора

**Цель:** сделать вкладку «Аналитика» полезной сразу после захода, без глубоких провалов.

**Что уже сделано:**
- Календарный месяц как базовый период.
- Совокупные метрики + разрез WB/Ozon/Яндекс.

**Что доработать в рамках Этапа 1:**
1. **Уточнение периодов (по желанию):**
   - Добавить селектор периода: «текущий месяц / прошлый месяц», но всегда по календарю, а не по скользящим 30 дням.

2. **UX‑добавки:**
   - Tooltips к KPI: что именно считается (например, WB: сумма `totalPrice/100` по заказам со статусом выкуплен).

**Заметка для следующего агента:**

### 5. Этап 2 — Аналитика по товарам

**Статус:** базовая реализация выполнена (февраль 2025).

**Сделано:**
- **Бэкенд:** `AnalyticsService.getProductStats(userId, from?, to?)` — агрегация заказов за период (по умолчанию текущий календарный месяц) по товарам и маркетплейсам. Ответ: массив `{ productId, title, article, imageUrl, stock, byMarketplace: { WILDBERRIES/OZON/YANDEX: { revenue, orders, delivered } }, totalRevenue, totalOrders, totalDelivered }`. Эндпоинт `GET /analytics/products?from=&to=` в `AnalyticsController`.
- **Next.js:** роут `GET /api/analytics/products` с проксированием на API и передачей токена.
- **Фронтенд:** блок «Аналитика по товарам» на `/dashboard/analytics` — таблица: Товар (фото + название, ссылка на карточку), колонки WB / Ozon (выручка, заказы), Итого (выручка, заказы), Выкуп %, Остаток. Сортировка по выручке (на бэкенде). При отсутствии данных — заглушка «Нет продаж за месяц».

**Цель (расширения):** показывать продавцу товары, которые тянут выручку, провисают по конверсии, вот‑вот закончатся или убыточны.

**План работ (оставшиеся пункты — по желанию):**

1. **Бэкенд: агрегаты по товарам**
   - Добавить сервис на API:
     - `GET /analytics/products?from=YYYY-MM-01&to=YYYY-MM-DD` (или аналогичный).
   - На выходе по каждому товару (canonical_sku):
     - выручка по WB / Ozon / Яндекс и в сумме;
     - заказы, выкуп, возвраты;
     - средняя цена продажи;
     - остатки (наш склад + склады WB/Ozon);
     - простой маржинальный расчёт (если есть себестоимость).
   - Источники:
     - наши заказы + маппинг на marketplace;
     - при необходимости — отчёты WB/Ozon (если не хватает данных в БД).

2. **Фронтенд: таблица товаров на вкладке «Аналитика»**
   - Блок «Аналитика по товарам» превращается в:
     - таблицу с колонками:
       - Товар (фото + название),
       - WB: выручка, заказы, выкуп (%),
       - Ozon: выручка, заказы, выкуп (%),
       - Яндекс: (когда появится),
       - Итого: выручка, заказы,
       - Возвраты (%),
       - Остатки / дней до обнуления.
     - фильтры:
       - маркетплейс, категория, бренд, только с продажами / без продаж.
     - сортировки:
       - по выручке, по росту/падению (vs предыдущий период), по марже.

3. **Карточка товара (подготовка к Этапу 3):**
   - При клике по товару:
     - навигация на `/dashboard/products/[id]?tab=analytics` или отдельный маршрут `/dashboard/analytics/product/[id]`.
   - Пока можно повторить те же агрегаты, что в таблице (без глубокой воронки).

### 6. Этап 3 — Воронки и поиск (WB в первую очередь)

**Цель:** понять, почему товар не продаётся:
- мало трафика,
- много просмотров, но мало заказов,
- проблема в ключах/поиске.

**План:**

1. **Интеграция WB Analytics API:**
   - Методы воронки карточки:
     - показы → просмотры → корзина → заказы → выкуп.
   - Методы по поисковым запросам:
     - по товару: какие запросы приводят к карточке;
     - по запросу: какие товары чаще всего кликают/покупают.

2. **Бэкенд: сервис `AnalyticsService`**
   - Кеширует ответы WB Analytics (например, на уровне дня).
   - Даёт API:
     - `GET /analytics/wb/products-funnel?from=&to=`,
     - `GET /analytics/wb/queries?productId=...`,
     - `GET /analytics/wb/queries/top`.

3. **Фронтенд: новые блоки на вкладке «Аналитика»**
   - Раздел «Поиск и видимость»:
     - топ‑запросы WB с показами, кликами, заказами;
     - список проблемных ключей: много показов, мало заказов.
   - На карточке товара:
     - мини‑воронка (бар‑чарт или шаги),
     - список ключей с CTR и конверсией.

### 7. Этап 4 — Рекомендации и алерты

**Цель:** перевод аналитики в конкретные действия.

**Подход (первая версия — без ML):**

1. **Набор правил (rule engine):**
   - Высокий трафик, низкая конверсия:
     - если просмотры >> медианы, а CR < порога:
       - рекомендация: «доработать карточку (фото/описание/отзывы)».
   - Низкий трафик, высокая конверсия:
     - «товар конвертит хорошо, но мало показов — имеет смысл включить рекламу / поднять ставки».
   - Остатки < N дней продаж:
     - «скоро закончится на WB/Ozon — пополните запасы».
   - Возвраты > X% с одинаковой причиной:
     - «проверьте описание/размерную сетку/качество».

2. **Бэкенд: `RecommendationsService`**
   - Потребляет агрегаты из Этапов 2–3.
   - Возвращает список рекомендаций:
     - тип (карточка, цена, остатки, сервис),
     - уровень важности,
     - связанные товары/маркетплейсы.

3. **Фронтенд: блок «Рекомендации» (`/dashboard/analytics`)**
   - Список карточек‑рекомендаций с коротким текстом и ссылками:
     - «Открыть товар»,
     - «Перейти к заказам», и т.п.

4. **В дальнейшем:**
   - Ежедневный дайджест на email / Telegram.
   - Расширение правил, возможно ML‑модели.

---

### 8. Этап 5 — Глубокая фин. аналитика и маржа (опционально)

**Цель:** показать не только выручку, но и **чистую прибыль** по товарам и площадкам.

**Требуется:**
- хранить себестоимость в HandySeller;
- подключить финансовые отчёты WB/Ozon/Яндекс:
  - комиссии, логистика, платное хранение, удержания.

**Результат:**
- Дашборд «Маржа и прибыльность»:
  - прибыль по товарам/категориям,
  - сравнение маржи WB vs Ozon vs Яндекс,
  - убыточные SKU.

### 9. Стиль и UX

Все новые экраны аналитики должны:
- использовать существующие компоненты `@handyseller/ui` (`Card`, `Button`, `Badge` и т.д.);
- следовать текущей сетке дашборда:
  - `container`, `space-y-6`, `grid gap-4 md:grid-cols-2 lg:grid-cols-4`;
- использовать иконки `lucide-react` в том же ключе, что `Главная`, `Заказы`, `Маркетплейсы`;
- писать тексты **простым русским языком** без перегруза англицизмами.
