# Анализ: Админ-панель HandySeller

## Текущее состояние

- **API**: NestJS, JWT-авторизация
- **Роли**: `User.role` = `USER` | `ADMIN` (уже есть, но не используется)
- **Guards**: только `JwtAuthGuard` — любой авторизованный пользователь видит только свои данные
- **Мультитенантность**: все сущности привязаны к `userId`

## Требования

1. **Клиенты** — управление пользователями системы
2. **Финансы** — выручка, подписки, платежи
3. **Отчёты** — аналитика по заказам, товарам, маркетплейсам
4. **Права доступа** — только для админов

---

## Варианты реализации прав

### Вариант 1: RBAC (Role-Based) — простой и надёжный ✅ Рекомендуется

**Суть**: роль `ADMIN` даёт доступ ко всем админ-разделам.

| Плюсы | Минусы |
|-------|--------|
| Уже есть `Role` в БД | Жёсткое разделение: USER / ADMIN |
| Минимум изменений | Нет тонкой настройки (например, «только отчёты») |
| Понятно и быстро внедрить | |
| Масштабируется добавлением новых ролей | |

**Реализация**:
- `RolesGuard` — проверка `user.role === 'ADMIN'`
- Эндпоинты `/api/admin/*` защищены `@UseGuards(JwtAuthGuard, RolesGuard('ADMIN'))`
- Фронт: раздел `/dashboard/admin/*` с проверкой роли после логина

---

### Вариант 2: RBAC + Permission

**Суть**: роли задают набор прав (`admin:users:read`, `admin:finance:write` и т.п.).

| Плюсы | Минусы |
|-------|--------|
| Гибкое разграничение | Сложнее в настройке и сопровождении |
| Можно делать «читатель отчётов» | Больше сущностей в БД |
| Подходит для нескольких админов с разными зонами ответственности | |

**Когда брать**: если планируется 5+ админов с разными зонами.

---

### Вариант 3: Отдельное приложение / поддомен

**Суть**: админка на отдельном домене (`admin.handyseller.ru`) или отдельном Next.js-приложении.

| Плюсы | Минусы |
|-------|--------|
| Чёткая изоляция | Дублирование кода, два деплоя |
| Можно вынести в отдельный проект | Усложнение инфраструктуры |

**Когда брать**: если админка сильно разрастётся и будет жить отдельно от основного продукта.

---

## Рекомендация: Вариант 1 (RBAC)

Для текущего этапа оптимален простой RBAC:

1. Уже есть `Role.ADMIN` в схеме.
2. Админов обычно мало — 1–3 человека.
3. При необходимости позже можно добавить Permission-слой поверх ролей.

## Архитектура (по варианту 1)

### API

```
api/
├── admin/
│   ├── admin.module.ts
│   ├── admin.controller.ts      # общий entry
│   ├── guards/
│   │   └── roles.guard.ts       # проверка role === ADMIN
│   ├── users/
│   │   └── admin-users.controller.ts   # GET /admin/users
│   ├── finance/
│   │   └── admin-finance.controller.ts # GET /admin/finance/summary
│   └── reports/
│       └── admin-reports.controller.ts # GET /admin/reports/orders
```

### Фронт

```
app/dashboard/
├── admin/                        # только для role=ADMIN
│   ├── layout.tsx                # AdminGuard + сайдбар
│   ├── page.tsx                  # дашборд админа
│   ├── users/
│   │   └── page.tsx              # клиенты
│   ├── finance/
│   │   └── page.tsx              # финансы
│   └── reports/
│       └── page.tsx              # отчёты
```

### Безопасность

1. **Роль всегда проверяется на бэкенде** — фронт только скрывает/показывает UI.
2. **JWT**: в токене можно хранить `role` (или подгружать при каждом запросе).
3. **Логирование**: действия админа пишем в `AuthAuditLog` или отдельную таблицу `AdminAuditLog`.
4. **Rate limit**: на `/api/admin/*` можно сделать строже, чем на обычные эндпоинты.

## Структура данных для админа

### Клиенты (Users)

- Список: id, email, name, role, createdAt, isActive, план подписки
- Детали: продукты, заказы, маркетплейсы
- Действия: блокировка (isActive), смена роли

### Финансы

- Выручка по пользователям (агрегация `Order.totalAmount`)
- Подписки: план, дата окончания
- Сводка: общая выручка, количество заказов

### Отчёты

- Заказы по периодам, маркетплейсам, статусам
- Товары: топ по продажам
- Новые регистрации по дням/неделям

---

## План внедрения (поэтапно)

### Фаза 1: База (≈1–2 дня)

1. `RolesGuard` с `@Roles('ADMIN')`
2. Первый админ: миграция или seed для задания `role = ADMIN` нужному пользователю
3. Эндпоинт `GET /api/admin/users` (список пользователей) с проверкой роли
4. Страница `/dashboard/admin` с защитой на фронте

### Фаза 2: Клиенты

1. `GET /api/admin/users` — пагинация, поиск, фильтры
2. `PATCH /api/admin/users/:id` — блокировка, смена роли
3. Страницы: список клиентов, карточка пользователя

### Фаза 3: Финансы

1. `GET /api/admin/finance/summary` — сводка
2. `GET /api/admin/finance/by-user` — выручка по пользователям
3. Страницы: общая сводка, детали по пользователям

### Фаза 4: Отчёты

1. `GET /api/admin/reports/orders` — заказы за период
2. `GET /api/admin/reports/products` — топ товаров
3. Экспорт (CSV/Excel) при необходимости

## Дополнительные меры

- **2FA для админов** — уже есть `twoFactorSecret` у `User`, можно требовать 2FA при role=ADMIN
- **IP whitelist** (опционально) — для особенно чувствительных админ-действий
- **Отдельная таблица AdminAuditLog** — кто, когда и что сделал в админке
