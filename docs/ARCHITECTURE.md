# Архитектура HandySeller: мультитенантность

## Принцип: изоляция данных по пользователю

Каждый клиент (пользователь) работает **только со своими данными** под **своими токенами** маркетплейсов.

### Механизм

1. **Авторизация**: пользователь входит под своим email/паролем, получает JWT с `userId` (sub).
2. **Все API-запросы** содержат `Authorization: Bearer <token>` — бэкенд извлекает `userId` из JWT.
3. **Сервисы** всегда фильтруют по `userId`:
   - `orders` → `where: { userId }`
   - `products` → `where: { userId }`
   - `marketplaceConnection` → `where: { userId }`
4. **Токены маркетплейсов** (WB, Ozon и т.д.) хранятся в `MarketplaceConnection` — по одной записи на пользователя и маркетплейс. Запросы к API маркетплейсов выполняются **только** через токены текущего пользователя.

### Пример: клиент nmanoilo@ya.ru

- Вход под своим логином/паролем → JWT с его `userId`
- Подключения WB/Ozon — его токены в его ЛК маркетплейса
- Заказы, товары, синхронизация — только его данные
- Управление маркетплейсами — только его кабинет

### Админ (например Rusinov@handyseller.ru)

- Роль `ADMIN` — доступ к `/admin` (пользователи, платежи)
- **Подключений к маркетплейсам у админа нет** — это ожидаемо
- Запросы статусов WB и т.п. вернут «Wildberries не подключён» — корректное поведение
- Дашборд, заказы, товары для админа пусты, если он не подключил маркетплейсы

### Итог

Запросы всегда идут **под токенами того пользователя, который вошёл**. Смешивания данных между пользователями не происходит.
